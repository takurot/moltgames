# Dev Task Notes: PR-14: Next.js フロント基盤

Started: 2026-03-24 22:18:05

## Research Findings

### Existing `apps/web` state

- `apps/web/` is a **stub TypeScript module**, not a Next.js app: `build: tsc`, no React/Next.js deps, single export file `src/index.ts`. Must be fully replaced.
- Current `package.json` has `"type": "module"` — Next.js config files (`next.config.ts`) work fine with ESM, but `next.config.js` must use `.mjs` extension or TS if `"type": "module"` is set.

### tsconfig / tooling constraints

- `tsconfig.base.json` sets `moduleResolution: Bundler`, `strict: true`, `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes` — all compatible with Next.js but JSX support (`"jsx": "preserve"`) and `"lib": ["dom", "dom.iterable", "ES2022"]` must be added in `apps/web/tsconfig.json`.
- The existing web `tsconfig.json` sets `"rootDir": "src"` and `"outDir": "dist"` — Next.js ignores these; they should be removed or overridden for the web package.
- Turbo task graph (`turbo.json`) expects `build`, `lint`, `typecheck`, `test:unit`, `test:integration`, `test:e2e` scripts — must be preserved in the new `package.json`.

### Firebase / Auth patterns

- Gateway defines two supported providers: `google.com` and `github.com` (`apps/gateway/src/auth/firebase-auth.ts:1`). Firebase Auth UI must offer both.
- Admin SDK (`firebase-admin`) is used only in gateway/engine. The web needs the **Firebase client SDK** (`firebase`), which is not yet installed anywhere in the monorepo.
- Firestore converters in `@moltgames/domain` use the admin SDK interface (`FirestoreConverter`). The client SDK uses the same `withConverter()` pattern — the domain converters can be adapted/reused on the frontend.

### Domain types to reuse

- `@moltgames/domain` exports all needed types: `Match`, `MatchStatus` (8 states), `User`, `AgentProfile`, `Rating`, `MatchParticipant`. Import directly in UI components.
- `MATCH_STATUSES`, `MATCH_TERMINAL_STATUSES`, `canTransitionMatchStatus` from `match-status.ts` are useful for displaying match state badges.

### Environment variables

- `NEXT_PUBLIC_API_BASE_URL` and `NEXT_PUBLIC_WS_BASE_URL` already declared in `.env.example`. Firebase client config vars (`NEXT_PUBLIC_FIREBASE_API_KEY`, etc.) need to be added.
- Firebase Auth emulator vars (`FIREBASE_AUTH_EMULATOR_HOST`) are set in `.env.example` — web should respect these in local dev via `connectAuthEmulator`.

### App Hosting

- No `apphosting.yaml` exists yet — must be created at repo root or `apps/web/`. Firebase App Hosting requires this file for Cloud Run deployment config.

### Testing

- All other packages use **vitest** (not Jest) for unit/integration/e2e. Web tests should follow the same pattern to stay consistent with the monorepo's `pnpm test:unit` turbo pipeline.

### Potential pitfalls

1. **`"type": "module"` + Next.js**: Use `next.config.ts` (TypeScript config) to avoid CJS/ESM conflicts.
2. **`noUncheckedIndexedAccess`**: React array renders like `items[0]` must use optional chaining or null checks.
3. **`exactOptionalPropertyTypes`**: Component props must distinguish `undefined` from absent fields explicitly.
4. **Firebase client vs admin converters**: Domain `FirestoreConverter<T, S>` interface is structurally compatible with client SDK's `FirestoreDataConverter` — verify `fromFirestore(snapshot)` signature matches `QueryDocumentSnapshot` vs admin `DocumentSnapshot`.
5. **No `apphosting.yaml` template in repo**: Check Firebase App Hosting docs for minimal required fields (`runConfig.cpu`, `runConfig.memoryMiB`, `runConfig.minInstances`).
