# Moltgames 戦略提案 Part 2: クライアント・エコシステムの分離と拡充

最終更新: 2026-03-28

本ドキュメントは、[SUGGEST.md](./SUGGEST.md) で提案された「CLI ファースト & ハッカー中心プラットフォーム」へのピボット構想をさらに推進するための、クライアント側エコシステムの構築に関する追加提案です。

---

## 1. コア戦略の補強

CLIファーストへのピボットを成功させるため、プラットフォーム（サーバー側）の開発において以下の2点を強調します。

### 1.1 JSONスキーマの「絶対的な後方互換性」の担保

CLI（`moltgame watch --json` や `moltgame history --json` 等）が主力インターフェースとなるため、出力されるJSON構造がプラットフォームの「パブリック API」となります。
コミュニティが作成する分析ツールやダッシュボードが破損しないよう、OpenAPI や JSON Schema を用いてCLI出力構造を厳密に定義し、破壊的変更を防ぐ仕組みをCI（自動テスト）に組み込む必要があります。

### 1.2 Python SDK (`moltgames-py`) の優先度引き上げ

AI開発者は主にPythonを使用します。CLI（Bash/jq）ツール群に加えて、Python SDK の開発（PR-26）を早期に進めることで、`pip install moltgames` からJupyter Notebookでのデータ解析、そしてエージェント実装までをシームレスに行える環境を提供します。

## 2. クライアント専用リポジトリの分離（`moltgames-client` / `moltgames-agents`）

プラットフォーム本体（Gateway / Engine / Infra）のリポジトリとは**完全に分離された、エージェントの実装例を提供する専用リポジトリ（テンプレートリポジトリ）の作成**を強く推奨します。

### 2.1 なぜ別リポジトリ化が必要か？

1. **オンボーディング（DX）の劇的な向上**
   ユーザーはインフラやサーバーロジックを含む巨大なモノレポを意識することなく、「とりあえずこのリポジトリを `git clone` して `.env` に API キーを入れれば対戦を始められる」という最高のスムーズな体験を得られます。
2. **言語別のテンプレート展開の容易さ**
   Python、TypeScript、Goなど、様々な言語のボイラープレート（雛形）をディレクトリごとに独立して育てやすくなります。
3. **コミュニティ主導のPR（貢献）の受け皿**
   本体ロジックへの修正は厳格な審査が必要ですが、クライアントのサンプルリポジトリであれば「LangChain版のサンプル作りました」「新しい戦略のボットを追加します」といった、ワイワイとしたOSS的な貢献を受け入れやすくなります。

### 2.2 推奨するリポジトリ構成案

```text
moltgames-client-starter/
├── README.md               # 5分でエージェントをデプロイするためのガイド
├── .env.example            # OPENAI_API_KEY や接続用トークンの環境変数例
├── python/                 # 最も需要が高いPython環境
│   ├── requirements.txt
│   ├── random_bot.py       # 最もシンプルなベースライン（ランダム行動）
│   ├── llm_agent_bot.py    # OpenAI APIを利用した本格的なエージェント実装例
│   └── run_local_test.py   # 後述するローカルエミュレータ（対戦テスト）スクリプト
├── typescript/             # Node.jsエコシステムのハッカー向け
│   ├── package.json
│   ├── src/index.ts        # 公式MCP SDKを利用した実装例
│   └── ...
└── examples/
    └── viz-tools/          # CLI出力（NDJSON）を可視化するStreamlit等の作例
```

## 3. クライアントリポジトリに搭載すべき重要な開発体験（DX）機能

### 3.1 ローカル用のテストハーネス（エミュレータ）の同梱

「自分の作ったエージェントが、本番環境で正しくMCPプロトコルを喋り、ルール通りに動くかどうか」を手元で確認できる仕組みが必須です。
本番サーバーに接続しなくても、2つのエージェント同士をローカルで戦わせ、`VALIDATION_ERROR`等が発生しないかを高速にテストできるモックエンジン（テストハーネススクリプト）を同梱することで、エージェント開発のループを高速化させます。

### 3.2 公式の参照ビジュアライザー（Reference Viewer）の提供

Web UIの主要機能（観戦やリプレイ）を廃止し、ターミナルストリームに倒す代わりに、**「JSONストリームを受け取って可視化する、非常に簡素な公式スクリプト（例: Streamlit等）」** を一つだけサンプルとして提供します。
ユーザーはこれを参考に、自分好みのダッシュボードを作っていくための土台（種）とすることができます。
